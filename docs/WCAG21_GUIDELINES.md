# WCAG 2.1 アクセシビリティガイドライン

Web Content Accessibility Guidelines (WCAG) 2.1は、Webコンテンツをより幅広い障害を持つ人々にとってアクセシブルにするための国際標準です。

## WCAG 2.1の4つの原則（POUR）

WCAG 2.1は4つの基本原則に基づいています：

### **P**erceivable（知覚可能）
情報とユーザーインターフェースコンポーネントは、ユーザーが知覚できる方法で提示される必要があります。

### **O**perable（操作可能）
ユーザーインターフェースコンポーネントとナビゲーションは操作可能である必要があります。

### **U**nderstandable（理解可能）
情報とユーザーインターフェースの操作は理解可能である必要があります。

### **R**obust（堅牢）
コンテンツは、支援技術を含む様々なユーザーエージェントによって確実に解釈できるよう、十分に堅牢である必要があります。

---

## 適合レベル

WCAG 2.1には3つの適合レベルがあります：

- **レベルA**：最低限の適合レベル（必須）
- **レベルAA**：推奨される適合レベル（多くの法規制で要求）
- **レベルAAA**：最高レベルの適合

**推奨**: 公開WebサイトはレベルAAへの適合を目指すべきです。

## 原則1: 知覚可能（Perceivable）

### 1.1 テキスト代替（Text Alternatives）

**ガイドライン**: すべての非テキストコンテンツには、テキスト代替を提供する。

#### 達成基準 1.1.1 非テキストコンテンツ（レベルA）

**要件**:
- すべての画像に適切な`alt`属性を提供
- 装飾的な画像には空の`alt=""`を使用
- SVG画像には`role="img"`と`aria-label`を提供
- フォームコントロールにはラベルを提供

**実装例**:
```html
<!-- 情報を伝える画像 -->
<img src="aws-diagram.png" alt="AWS VPCとDirect Connectの接続図">

<!-- 装飾的な画像 -->
<img src="decoration.png" alt="">

<!-- SVG画像 -->
<svg role="img" aria-label="AWSアーキテクチャ図">
  <!-- SVG content -->
</svg>
```

---

### 1.2 時間依存メディア（Time-based Media）

**ガイドライン**: 時間依存メディア（動画、音声）には代替を提供する。

#### 達成基準 1.2.1 音声のみ及び映像のみ（収録済み）（レベルA）
- 音声のみのコンテンツには書き起こしテキストを提供
- 映像のみのコンテンツには音声トラックまたは書き起こしを提供

**このプロジェクトでの関連性**: 現在、動画・音声コンテンツはありませんが、将来追加する場合は考慮が必要です。

---

### 1.3 適応可能（Adaptable）

**ガイドライン**: 情報や構造を失うことなく、様々な方法で提示できるコンテンツを作成する。

#### 達成基準 1.3.1 情報及び関係性（レベルA）

**要件**:
- セマンティックHTMLを使用（`<header>`, `<nav>`, `<main>`, `<article>`, `<section>`, `<footer>`）
- 見出しレベルを適切に使用（h1 → h2 → h3、飛ばさない）
- リストには`<ul>`, `<ol>`, `<li>`を使用
- テーブルには`<table>`, `<th>`, `<td>`を適切に使用
- フォームには`<label>`とフォームコントロールを関連付け

**実装例**:
```html
<!-- セマンティックHTML -->
<header>
  <h1>AWS SAP学習リソース</h1>
</header>

<nav aria-label="メインナビゲーション">
  <ul>
    <li><a href="#networking">ネットワーキング</a></li>
  </ul>
</nav>

<main>
  <article>
    <h2>ネットワーキング</h2>
    <section>
      <h3>VPC基礎</h3>
    </section>
  </article>
</main>

<!-- フォームラベル -->
<label for="search">検索:</label>
<input type="text" id="search" name="search">
```

#### 達成基準 1.3.2 意味のある順序（レベルA）

**要件**:
- コンテンツの読み上げ順序は意味のある順序である必要がある
- CSSで視覚的な順序を変更しても、DOMの順序は論理的であるべき

#### 達成基準 1.3.3 感覚的な特徴（レベルA）

**要件**:
- 形状、大きさ、視覚的な位置、方向、音だけに依存した指示を避ける
- 例: ❌「右側のボタンをクリック」→ ✅「次へボタンをクリック」

#### 達成基準 1.3.4 表示の向き（レベルAA）- WCAG 2.1新規

**要件**:
- コンテンツは縦向きと横向きの両方の表示に対応すべき
- 特定の向きに制限しない（例外：銀行小切手、ピアノアプリなど）

#### 達成基準 1.3.5 入力目的の特定（レベルAA）- WCAG 2.1新規

**要件**:
- フォーム入力フィールドには`autocomplete`属性を使用
- ユーザー情報（名前、メールアドレス、電話番号など）の入力フィールドで特に重要

**実装例**:
```html
<input type="text" name="name" autocomplete="name">
<input type="email" name="email" autocomplete="email">
```

### 1.4 判別可能（Distinguishable）

**ガイドライン**: ユーザーがコンテンツを見たり聞いたりしやすくする。

#### 達成基準 1.4.1 色の使用（レベルA）

**要件**:
- 色だけで情報を伝えない
- 例: エラーメッセージは赤色だけでなく、アイコンやテキストでも示す

**実装例**:
```html
<!-- ❌ 色だけで示す -->
<span style="color: red;">必須</span>

<!-- ✅ 色とテキストで示す -->
<span style="color: red;">* 必須項目</span>

<!-- ✅ 色とアイコンで示す -->
<span style="color: red;" aria-label="必須">⚠️</span>
```

#### 達成基準 1.4.2 音声の制御（レベルA）

**要件**:
- 自動再生される音声は3秒以内に停止、または一時停止/音量調整メカニズムを提供

**このプロジェクトでの関連性**: 現在、自動再生音声はありません。

#### 達成基準 1.4.3 コントラスト（最低限）（レベルAA）

**要件**:
- テキストと背景のコントラスト比は**最低4.5:1**（大きいテキストは3:1）
- 大きいテキスト：18pt以上、または14pt太字以上

**AWS SAP学習サイトのカラーパレット検証**:
```
Primary Dark: #232F3E (AWS dark blue)
Accent Orange: #FF9900 (AWS orange)
Text Gray: #374151 (dark gray)
Background: #F9FAFB (light gray)
Border: #E5E7EB (medium gray)
```

**コントラスト比計算が必要**:
- `#232F3E`（濃紺）on `#F9FAFB`（薄灰） → 検証必要
- `#FF9900`（オレンジ）on `#FFFFFF`（白） → 検証必要
- `#374151`（濃灰）on `#F9FAFB`（薄灰） → 検証必要

**推奨ツール**: [WebAIM Contrast Checker](https://webaim.org/resources/contrastchecker/)

#### 達成基準 1.4.4 テキストのサイズ変更（レベルAA）

**要件**:
- テキストは支援技術なしで200%まで拡大可能
- コンテンツや機能を失わない

**実装のベストプラクティス**:
- `px`ではなく`rem`または`em`を使用
- `font-size: 16px`を基準に相対サイズを設定

#### 達成基準 1.4.5 文字画像（レベルAA）

**要件**:
- テキストは画像ではなくテキストで実装
- 例外：ロゴ、装飾的な文字画像

**このプロジェクトでの関連性**: SVG図内のテキストは画像扱いになる可能性があるため、重要な情報はHTML本文にも記載すべき。

#### 達成基準 1.4.10 リフロー（レベルAA）- WCAG 2.1新規

**要件**:
- 横スクロールなしで、320px幅（モバイル）でコンテンツが表示できる
- レスポンシブデザインの実装

**実装チェック**:
- 現在のブレークポイント: Mobile < 768px
- 320px幅でのテスト必須

#### 達成基準 1.4.11 非テキストのコントラスト（レベルAA）- WCAG 2.1新規

**要件**:
- UIコンポーネント（ボタン、フォーム入力枠）と隣接色のコントラスト比：**最低3:1**
- グラフィカルオブジェクト（アイコン、図の重要部分）と背景のコントラスト比：**最低3:1**

**このプロジェクトでの関連性**:
- ボタンのボーダー、フォーム入力枠のボーダーのコントラスト
- SVG図内の線や図形のコントラスト

#### 達成基準 1.4.12 テキストの間隔（レベルAA）- WCAG 2.1新規

**要件**:
- ユーザーが以下のスタイルを適用しても、コンテンツや機能を失わない：
  - 行の高さ：フォントサイズの1.5倍以上
  - 段落の間隔：フォントサイズの2倍以上
  - 文字間隔：フォントサイズの0.12倍以上
  - 単語間隔：フォントサイズの0.16倍以上

**実装のベストプラクティス**:
```css
body {
  line-height: 1.5; /* 最低1.5 */
}

p {
  margin-bottom: 2em; /* 段落間隔 */
}
```

#### 達成基準 1.4.13 ホバーまたはフォーカスで表示されるコンテンツ（レベルAA）- WCAG 2.1新規

**要件**:
- ツールチップやホバーメニューは：
  1. **解除可能**：Escキーで閉じられる
  2. **ホバー可能**：ポインタをツールチップ上に移動できる
  3. **持続的**：ユーザーが解除するまで表示される

**このプロジェクトでの関連性**: 現在ツールチップ機能はないが、将来追加する場合は考慮が必要。

## 原則2: 操作可能（Operable）

### 2.1 キーボードアクセス可能（Keyboard Accessible）

**ガイドライン**: すべての機能をキーボードから利用可能にする。

#### 達成基準 2.1.1 キーボード（レベルA）

**要件**:
- すべての機能はキーボードのみで操作可能
- マウスがなくても完全に利用できる

**実装チェック**:
- すべてのボタン、リンク、フォームがTabキーでフォーカス可能
- Enterキー/Spaceキーで実行可能
- カスタムウィジェット（アコーディオン、モーダルなど）もキーボード操作可能

**実装例**:
```html
<!-- インタラクティブ要素は適切なHTML要素を使用 -->
<button onclick="toggleCategory('networking')">カテゴリを開く</button>

<!-- ❌ divにonclickは避ける -->
<div onclick="toggleCategory('networking')">カテゴリを開く</div>

<!-- ✅ divを使う場合はtabindexとキーボードイベントを追加 -->
<div role="button" tabindex="0" onclick="toggleCategory('networking')"
     onkeydown="if(event.key==='Enter'||event.key===' '){toggleCategory('networking')}">
  カテゴリを開く
</div>
```

#### 達成基準 2.1.2 キーボードトラップなし（レベルA）

**要件**:
- フォーカスがトラップされない（キーボードで脱出できる）
- モーダルダイアログを開いた場合、Escキーで閉じられる

#### 達成基準 2.1.4 文字キーのショートカット（レベルA）- WCAG 2.1新規

**要件**:
- 単一文字キーのショートカットがある場合、以下のいずれかを満たす：
  1. オフにできる
  2. 再マッピングできる
  3. フォーカス時のみ有効

**このプロジェクトでの関連性**: 現在、文字キーショートカットはありませんが、検索機能に「/」キーなどを追加する場合は考慮が必要。

---

### 2.2 十分な時間（Enough Time）

**ガイドライン**: ユーザーがコンテンツを読み、使用するために十分な時間を提供する。

#### 達成基準 2.2.1 タイミング調整可能（レベルA）

**要件**:
- 制限時間がある場合、ユーザーが無効化、調整、または延長できる
- 例外：リアルタイムイベント、オークション、20時間以上の制限

**このプロジェクトでの関連性**: クイズに制限時間はないため、現在は該当なし。

#### 達成基準 2.2.2 一時停止、停止、非表示（レベルA）

**要件**:
- 自動で動く、点滅する、スクロールするコンテンツ（5秒以上）は一時停止可能

**このプロジェクトでの関連性**: 現在、自動アニメーションはありませんが、カルーセルやスライダーを追加する場合は考慮が必要。

---

### 2.3 発作と身体反応（Seizures and Physical Reactions）

**ガイドライン**: 発作を引き起こすことが知られている方法でコンテンツを設計しない。

#### 達成基準 2.3.1 3回の閃光、又は閾値以下（レベルA）

**要件**:
- 1秒間に3回以上の閃光を発するコンテンツは含めない

**このプロジェクトでの関連性**: 現在該当なし。アニメーション追加時は注意。

---

### 2.4 ナビゲート可能（Navigable）

**ガイドライン**: ユーザーがナビゲートし、コンテンツを見つけ、現在位置を確認できるようにする。

#### 達成基準 2.4.1 ブロックスキップ（レベルA）

**要件**:
- 繰り返されるコンテンツブロック（ヘッダー、ナビゲーション）をスキップする仕組みを提供
- 実装：「メインコンテンツへスキップ」リンク

**実装例**:
```html
<body>
  <a href="#main" class="skip-link">メインコンテンツへスキップ</a>

  <header>...</header>
  <nav>...</nav>

  <main id="main">
    <!-- メインコンテンツ -->
  </main>
</body>

<style>
.skip-link {
  position: absolute;
  top: -40px;
  left: 0;
  background: #000;
  color: #fff;
  padding: 8px;
  z-index: 100;
}

.skip-link:focus {
  top: 0;
}
</style>
```

#### 達成基準 2.4.2 ページタイトル（レベルA）

**要件**:
- すべてのページに説明的で一意な`<title>`を提供

**実装例**:
```html
<title>ネットワーキング - AWS Direct Connect完全ガイド | AWS SAP学習リソース</title>
```

**推奨フォーマット**: `[ページ内容] - [セクション名] | [サイト名]`

#### 達成基準 2.4.3 フォーカス順序（レベルA）

**要件**:
- Tabキーでのフォーカス順序は論理的で意味のある順序

**実装チェック**:
- DOMの順序 = タブ順序
- CSSで視覚的な順序を変更してもDOM順序は論理的

#### 達成基準 2.4.4 リンクの目的（コンテキスト内）（レベルA）

**要件**:
- リンクの目的がリンクテキストから、またはリンクのコンテキストから判断できる
- ❌「こちら」「詳細」だけのリンクは避ける

**実装例**:
```html
<!-- ❌ 避けるべき -->
<a href="vpc.html">こちら</a>をクリックしてVPCについて学ぶ

<!-- ✅ 推奨 -->
<a href="vpc.html">VPCの基礎ガイド</a>で学習を開始

<!-- ✅ aria-labelで補足 -->
<a href="vpc.html" aria-label="VPCの基礎ガイドを読む">詳細</a>
```

#### 達成基準 2.4.5 複数の方法（レベルAA）

**要件**:
- コンテンツを見つけるための複数の方法を提供
- 例：ナビゲーションメニュー、検索機能、サイトマップ、目次

**このプロジェクトでの実装状況**:
- ✅ サイドバーナビゲーション
- ✅ 検索機能
- ✅ カテゴリクイックナビゲーション
- ✅ table-of-contents.html
- ✅ パンくずリスト

#### 達成基準 2.4.6 見出し及びラベル（レベルAA）

**要件**:
- 見出しとラベルはトピックまたは目的を説明する
- 見出しは内容を明確に示す

**実装チェック**:
- h1はページタイトル
- h2以降は論理的な階層構造
- フォーム入力には明確なラベル

#### 達成基準 2.4.7 フォーカスの可視化（レベルAA）

**要件**:
- キーボードフォーカスインジケーターが視覚的に見える

**実装のベストプラクティス**:
```css
/* デフォルトのoutlineを削除しない */
/* ❌ 避けるべき */
*:focus {
  outline: none;
}

/* ✅ カスタムフォーカススタイルを提供する場合 */
button:focus,
a:focus {
  outline: 2px solid #FF9900; /* AWS orange */
  outline-offset: 2px;
}
```

### 2.5 入力モダリティ（Input Modalities）- WCAG 2.1新規

**ガイドライン**: キーボード以外の様々な入力方法で機能を操作できるようにする。

#### 達成基準 2.5.1 ポインタのジェスチャー（レベルA）

**要件**:
- マルチポイントジェスチャ（ピンチズーム）や経路ベースのジェスチャは、シングルポイントでも操作可能

**このプロジェクトでの関連性**: 現在、複雑なジェスチャは使用していません。

#### 達成基準 2.5.2 ポインタのキャンセル（レベルA）

**要件**:
- シングルポイントで操作する機能は、以下のいずれかを満たす：
  1. downイベントではなくupイベントで実行
  2. アボート機能がある
  3. upイベントで取り消し可能

**実装のベストプラクティス**:
```javascript
// ✅ clickイベント（upイベントベース）を使用
button.addEventListener('click', handleClick);

// ❌ mousedownやtouchstartは避ける（特別な理由がない限り）
button.addEventListener('mousedown', handleClick);
```

#### 達成基準 2.5.3 ラベルを含む名前（レベルA）

**要件**:
- 視覚的なラベルテキストは、アクセシブルな名前（aria-labelなど）に含まれる

**実装例**:
```html
<!-- ✅ 視覚的ラベルとaria-labelが一致 -->
<button aria-label="検索を実行">検索</button>

<!-- ❌ 視覚的ラベルとaria-labelが不一致（音声入力で混乱） -->
<button aria-label="サーチ">検索</button>
```

#### 達成基準 2.5.4 動きによる起動（レベルA）

**要件**:
- デバイスの動き（振る、傾けるなど）で操作する機能は、UIコンポーネントでも操作可能
- 動き検出を無効化できる

**このプロジェクトでの関連性**: 現在、デバイスの動きによる機能はありません。

## 原則3: 理解可能（Understandable）

### 3.1 読みやすさ（Readable）

**ガイドライン**: テキストコンテンツを読みやすく理解可能にする。

#### 達成基準 3.1.1 ページの言語（レベルA）

**要件**:
- `<html>`タグに`lang`属性を指定

**実装例**:
```html
<!DOCTYPE html>
<html lang="ja">
```

**このプロジェクトでの実装状況**: ✅ すでに実装済み（CODING_STANDARDS.mdで要求）

#### 達成基準 3.1.2 一部分の言語（レベルAA）

**要件**:
- ページの一部が異なる言語の場合、`lang`属性で示す

**実装例**:
```html
<p>AWSの<span lang="en">Virtual Private Cloud (VPC)</span>について学習します。</p>
```

**このプロジェクトでの関連性**: AWS用語は英語なので、厳密には`lang="en"`を追加すべきだが、実用上は省略可能。

---

### 3.2 予測可能（Predictable）

**ガイドライン**: Webページの表示と動作を予測可能にする。

#### 達成基準 3.2.1 フォーカス時（レベルA）

**要件**:
- コンポーネントがフォーカスを受け取っただけで、コンテキストの変化を引き起こさない
- 例：フォーカスしただけで新しいウィンドウを開かない

#### 達成基準 3.2.2 入力時（レベルA）

**要件**:
- 設定の変更が自動的にコンテキストの変化を引き起こさない（事前に説明がない限り）
- 例：プルダウンメニューを選択しただけでページ遷移しない

**実装のベストプラクティス**:
```html
<!-- ✅ 送信ボタンを提供 -->
<select id="category">
  <option>ネットワーキング</option>
  <option>セキュリティ</option>
</select>
<button>表示</button>

<!-- ❌ onchangeで自動遷移は避ける -->
<select onchange="location.href=this.value">
  <option value="networking.html">ネットワーキング</option>
</select>
```

#### 達成基準 3.2.3 一貫したナビゲーション（レベルAA）

**要件**:
- 複数のページで繰り返されるナビゲーションは一貫した順序で表示

**このプロジェクトでの関連性**: すべてのページで同じナビゲーション構造を維持する必要があります。

#### 達成基準 3.2.4 一貫した識別（レベルAA）

**要件**:
- 同じ機能を持つコンポーネントは一貫して識別される
- 例：すべてのページで検索アイコンは同じデザイン

---

### 3.3 入力支援（Input Assistance）

**ガイドライン**: ユーザーが間違いを回避し、修正できるよう支援する。

#### 達成基準 3.3.1 エラーの特定（レベルA）

**要件**:
- 入力エラーが自動的に検出された場合、エラー項目を特定し、テキストで説明

**実装例**:
```html
<!-- フォーム検証 -->
<form>
  <label for="email">メールアドレス *</label>
  <input type="email" id="email" required aria-describedby="email-error">
  <span id="email-error" role="alert" style="color: red; display: none;">
    ⚠️ 有効なメールアドレスを入力してください
  </span>
</form>
```

#### 達成基準 3.3.2 ラベル又は説明（レベルA）

**要件**:
- ユーザー入力が必要な場合、ラベルまたは説明を提供

**実装例**:
```html
<label for="name">氏名 <span aria-label="必須項目">*</span></label>
<input type="text" id="name" required>
```

#### 達成基準 3.3.3 エラー修正の提案（レベルAA）

**要件**:
- 入力エラーが検出され、修正方法が分かっている場合、提案を提供

**実装例**:
```html
<span role="alert">
  メールアドレスに「@」が含まれていません。例: user@example.com
</span>
```

#### 達成基準 3.3.4 エラー回避（法的、金融、データ）（レベルAA）

**要件**:
- 法的義務、金融取引、データの変更・削除を行うページでは、以下のいずれかを満たす：
  1. 取り消し可能
  2. 入力内容をチェックし、エラー修正の機会を提供
  3. 確認画面を提供

**このプロジェクトでの関連性**: 現在該当なし（クイズ結果の送信などはない）。

## 原則4: 堅牢（Robust）

### 4.1 互換性（Compatible）

**ガイドライン**: 支援技術を含む現在および将来のユーザーエージェントとの互換性を最大化する。

#### 達成基準 4.1.1 構文解析（レベルA）

**要件**:
- HTMLは仕様に準拠（開始タグと終了タグが適切、IDの重複がない、属性が適切など）
- **W3C HTML Validator**で検証

**このプロジェクトでの実装状況**: ✅ DEVELOPMENT_GUIDE.mdでW3C検証を必須としている

#### 達成基準 4.1.2 名前（name）、役割（role）及び値（value）（レベルA）

**要件**:
- すべてのUIコンポーネントには、名前と役割をプログラムで決定でき、状態・プロパティ・値が設定可能

**実装例**:
```html
<!-- 標準HTML要素は自動的に役割を持つ -->
<button>送信</button> <!-- role="button"が暗黙的 -->

<!-- カスタムウィジェットにはARIAを使用 -->
<div role="button" tabindex="0" aria-pressed="false">
  トグルボタン
</div>

<!-- 展開可能セクション -->
<button aria-expanded="false" aria-controls="panel1">
  セクションを展開
</button>
<div id="panel1" hidden>
  コンテンツ
</div>
```

#### 達成基準 4.1.3 ステータスメッセージ（レベルAA）- WCAG 2.1新規

**要件**:
- ステータスメッセージは支援技術に通知される（フォーカスを移動せずに）
- 実装：ARIA live regions (`role="status"`, `role="alert"`, `aria-live`)

**実装例**:
```html
<!-- 検索結果の通知 -->
<div role="status" aria-live="polite">
  検索結果: 25件見つかりました
</div>

<!-- エラーメッセージ -->
<div role="alert" aria-live="assertive">
  ⚠️ ファイルのアップロードに失敗しました
</div>

<!-- 進捗状況 -->
<div role="status" aria-live="polite">
  読み込み中... 75%完了
</div>
```

**aria-liveの値**:
- `polite`: ユーザーの作業が終わってから通知
- `assertive`: 即座に通知（エラーなど緊急性が高い場合）

## クイックリファレンス：実装優先順位

### 🔴 最優先（レベルA必須項目）
